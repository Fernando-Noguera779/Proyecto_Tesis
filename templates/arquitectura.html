1. Arquitectura del Sistema

El diseño se basa en un modelo de n-capas donde el Sistema Web actúa como intermediario (middleware) entre el usuario final y el clúster de cómputo.

A. Capa de Aplicación
Sistema Web
  • Frontend: Interfaz para enviar trabajos (jobs), monitorear el clúster y ver resultados.
  • Backend: Lógica para validación, gestión de proyectos y comunicación vía APIs o SSH.
  • Base de Datos Web (PostgreSQL): Almacena perfiles, roles, metadatos, historial y logs.
B. Capa de Gestión
Slurm Orchestrator
  • Slurmctld (Controller): El cerebro que gestiona la cola de trabajos.
  • Slurmdbd (Daemon): Interfaz entre Slurm y su base de datos para contabilidad.
  • Base de Datos Slurm (MySQL): Datos de contabilidad, límites de usuarios y tiempos.
C. Capa de Infraestructura
Hardware (HCP)
  • Sistema HCP: Monitoreo de bajo nivel (temperatura, energía, estado físico).
  • Nodos de Cómputo: Los servidores físicos donde se ejecutan las tareas.

2. Flujo de Datos

1. Solicitud

El usuario envía un script de cálculo desde el Sistema Web.

2. Registro

El Backend guarda la metadata en la Base de Datos Web.

3. Envío

El Backend se comunica con el Slurmctld para encolar el trabajo.

4. Ejecución

Slurm asigna el trabajo a los Nodos basándose en disponibilidad.

5. Monitoreo HCP

Envía alertas al Sistema Web si un nodo físico presenta fallas.

6. Contabilidad

Slurmdbd actualiza el consumo al terminar en BD Slurm.

3. Modelo de Entidad-Relación

Base de Datos Web (Gestión Académica)
Tabla Descripción
Usuarios ID, Nombre, Rol (Admin/Investigador), Credenciales.
Proyectos ID, Nombre, Descripción, Presupuesto de horas.
Solicitudes_Jobs ID, ID_Usuario, ID_Proyecto, Script_Path, Estado (En espera, Ejecutando, Fallido).
Base de Datos Slurm (Cómputo)
Tabla Descripción
Assoc_Table Relación entre usuarios, cuentas de Slurm y particiones.
Job_Table Registros de tiempo de inicio, fin, nodos utilizados y memoria consumida.

4. Consideraciones de Integración

Sincronización

Es vital que los usuarios creados en el Sistema Web tengan una correspondencia en Slurm (vía comandos sacctmgr) para que los permisos sean consistentes.

Seguridad

El Sistema Web no debe ejecutar comandos como root. Se recomienda el uso de un Nodo de Login dedicado o una API REST que actúe como puente seguro.

Automatización

PM2 en el backend asegura que el puente Web-Slurm esté siempre activo; el HCP garantiza que no se envíen trabajos a hardware degradado.
